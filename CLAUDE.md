# PROJECT_OVERVIEW

## 1. 서비스 목적
- 문제 정의
  - Next.js 기반 웹 어드민 서비스를 구축한다.
  - Codex가 기능정의서를 단일 기준으로 읽고 코드 구현을 수행할 수 있는 기준 문서를 제공한다.
  - Claude와 Codex가 동일 기준으로 요구사항을 해석하도록 문서 구조를 표준화한다.
- 주요 사용자
  - 서비스 기획자
  - 프론트엔드 개발자
  - 백엔드 개발자
  - QA 담당자
  - 운영자

## 2. 범위
- 이번 구현 범위
  - 기능정의서 기반 구현용 공통 기준 문서 작성
  - 화면 / 기능 / 상태 / API / 예외 정의용 문서 템플릿 제공
  - Claude와 Codex 협업 시 해석 기준 통일
- 제외 범위
  - 기능정의서에 없는 신규 기능 정의
  - 정책 미정 항목에 대한 임의 결정
  - 디자인 시스템 상세 시안 정의
  - 백엔드 실제 API 스펙 확정

## 3. 기술 스택
- 필수
  - Next.js
  - TypeScript
- 문서 협업
  - Markdown
  - Google Sheets 또는 Notion
- 구현 연계
  - React
  - API 연동 레이어
  - Mock 데이터 레이어

# FEATURE_SPEC

## 공통 구현 원칙

### 1. 목적
- 기능정의서에 명시된 기능만 구현 대상으로 삼는다.
- 기능정의서에 없는 항목은 구현하지 않고 `OPEN_QUESTIONS`에 기록한다.

### 2. 화면 구성
- 현재 확정 화면
  - 대시보드
  - 운영자 관리
  - 문서 관리
  - 로그
  - 피드백

### 3. 기능 상세
- 사용자 액션
  - 메뉴 진입
  - 목록 조회
  - 상세 조회
  - 수정
  - 저장
- 시스템 동작
  - 기능정의서에 정의된 항목만 노출
  - 정의되지 않은 액션은 UI에 노출하지 않음
  - 상태값, 권한, 예외는 기능정의서 기준으로만 처리

### 4. 데이터 구조
| 필드명 | 타입 | 설명 |
|--------|------|------|
| 미정 | 미정 | 기능정의서 필요 |

### 5. 상태값
- 공통 UI 상태만 허용
  - `LOADING`
  - `EMPTY`
  - `ERROR`
- 도메인 상태값
  - 기능정의서 미제공

### 6. 정렬 / 필터 / 검색
- 정렬 기준
  - 기능정의서 미제공
- 필터 조건
  - 기능정의서 미제공
- 검색 기준
  - 기능정의서 미제공

### 7. 예외 처리
- 데이터 없음
  - 빈 상태 화면 노출
  - 사용자 메시지: `조회된 데이터가 없습니다.`
- 권한 없음
  - 접근 차단
  - 사용자 메시지: `접근 권한이 없습니다.`
- 에러 발생
  - 오류 상태 화면 또는 토스트 노출
  - 사용자 메시지: `일시적인 오류가 발생했습니다. 다시 시도해주세요.`

### 8. 권한 정책
- 기능별 권한 정의는 기능정의서 미제공
- 미정 권한은 구현 금지

### 9. API 정의
- 공통 원칙
  - 실제 API 스펙이 없으면 mock 데이터로 분리 구현
  - API 인터페이스는 기능 단위로 분리
- 확정 API
  - 없음
- request / response 구조
  - 기능정의서 미제공

# OPEN_QUESTIONS

- 기능정의서 원문 링크 또는 문서 ID가 필요합니다.
- 각 메뉴별 상세 기능 목록이 필요합니다.
- 대시보드에 표시할 지표 항목이 필요합니다.
- 운영자 관리의 생성/삭제 가능 여부가 필요합니다.
- 운영자 권한 레벨 종류와 권한별 접근 범위가 필요합니다.
- 문서 관리의 문서 타입, 문서 상태 전이 규칙이 필요합니다.
- 로그의 카테고리 분류 기준이 필요합니다.
- 로그 보관 기간과 다운로드 기능 필요 여부가 필요합니다.
- 피드백 처리 상태 전이 규칙이 필요합니다.
- 각 목록의 페이지네이션 방식이 필요합니다.
- 정렬 기본값과 다중 정렬 허용 여부가 필요합니다.
- 필터 조건의 정확한 항목과 선택 방식이 필요합니다.
- 검색 대상 필드와 검색 방식(부분 일치 / 완전 일치)이 필요합니다.
- 각 기능별 접근 권한과 역할명이 필요합니다.
- 실제 API 존재 여부가 필요합니다.
- API base URL, 인증 방식, 에러 코드 규격이 필요합니다.
- request / response 상세 스키마가 필요합니다.
- 저장 성공 / 실패 후 후속 UX 정책이 필요합니다.
- UI 디자인 시스템 또는 레이아웃 기준 문서가 필요합니다.
- 기간 선택(`일/주/월`)별 실제 API 응답 스키마가 필요합니다.
- 접속자 수 / 총 문의 수 / 지식 응답 비중의 실데이터 집계 기준 응답 포맷이 필요합니다.

> 기능정의서에 없는 항목이 필요합니다  
> 이유: 화면별 기능, 상태, API, 권한이 확정되지 않으면 Codex가 구현 범위를 고정할 수 없습니다.  
> 영향 범위: FEATURE_SPEC, API 정의, 상태값, 예외 처리, 권한 정책, 화면 구현 전반  
> 제안 기능: 기능정의서 원문 공유 또는 메뉴별 상세 요구사항 확정  
> 기능정의서 수정 후 진행할까요?

# CODEX_EXECUTION_RULES

## 1. 구현 입력 기준
- Codex는 아래 문서를 우선순위 기준으로 읽는다.
  - 1순위: 기능정의서 원문
  - 2순위: `CLAUDE.md`
  - 3순위: 사용자의 추가 지시
- 우선순위 충돌 시 기능정의서를 기준으로 판단한다.

## 2. 구현 가능 조건
- 아래 조건이 모두 충족될 때만 구현한다.
  - 기능명 존재
  - 화면 존재
  - 사용자 액션 존재
  - 상태값 존재 또는 상태 미사용이 명시됨
  - API 또는 mock 기준 존재
  - 권한 정책 존재 또는 권한 미사용이 명시됨

## 3. 구현 금지 조건
- 아래 항목 중 하나라도 해당하면 구현하지 않는다.
  - 기능정의서에 없는 기능
  - 기능정의서에 없는 버튼
  - 기능정의서에 없는 상태값
  - 기능정의서에 없는 필터
  - 기능정의서에 없는 API 호출
  - 기능정의서에 없는 권한 처리
  - 기능정의서에 없는 데이터 필드

## 4. 미정 항목 처리 규칙
- 미정 항목은 추정하지 않는다.
- 미정 항목은 반드시 `OPEN_QUESTIONS`에 추가한다.
- 미정 항목이 구현 필수 조건이면 작업 상태를 `blocked`로 기록한다.
- 미정 항목이 비필수 조건이면 해당 기능 범위를 제외하고 진행 가능 여부를 먼저 문서에 기록한다.

## 5. 코드 생성 규칙
- Next.js App Router 기준으로 구현한다.
- TypeScript 필수
- 기능 단위 폴더 분리
- UI 컴포넌트 / 비즈니스 로직 / API 인터페이스 분리
- mock 데이터와 실제 API 분리
- 상태값은 상수 또는 타입으로 분리
- 하드코딩 금지 대상
  - 역할명
  - 상태값
  - API URL
  - 필터 옵션
  - 정렬 기준

## 6. 화면 구현 규칙
- 공통 화면 구조
  - 목록
  - 상세
  - 수정
  - 저장
- 읽기 전용 화면은 수정/저장 UI를 만들지 않는다.
- 공통 UI 상태를 반드시 포함한다.
  - 로딩
  - 빈 상태
  - 에러
- 사용자 메시지는 문서에 정의된 문구만 사용한다.
- 문구가 없으면 `OPEN_QUESTIONS`로 분리한다.

## 7. API 구현 규칙
- 실제 API 스펙이 없으면 mock 기반으로 구현한다.
- 실제 API 스펙이 있어도 request / response 필드가 없으면 연동 구현하지 않는다.
- 에러 응답 규격이 없으면 공통 오류 처리만 적용하고 세부 분기 금지
- 인증 방식이 없으면 보호 라우트 구현 금지

## 8. 권한 구현 규칙
- 권한 정의가 없으면 메뉴 숨김/노출 로직 구현 금지
- 권한 정의가 일부만 있으면 정의된 범위만 구현
- 상세 권한 미정 시 관리자 전체 권한으로 추정하지 않는다

## 9. 작업 결과 기록 규칙
- Codex는 작업 종료 시 아래 형식으로 기록한다.
  - 작업 목적
  - 이번 턴에서 한 일
  - 변경 파일
  - 확인 결과
  - 다음 AI가 바로 할 일
  - 리스크 / 확인 필요사항

## 10. 완료 기준
- 아래 조건 충족 시 완료로 판단한다.
  - 기능정의서와 구현 내용 일치
  - 불필요한 기능 추가 없음
  - 상태/예외/UI 메시지 반영 완료
  - mock 또는 API 연동 기준 반영 완료
  - 미정 항목은 `OPEN_QUESTIONS`에 기록 완료
